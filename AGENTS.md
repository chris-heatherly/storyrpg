# StoryRPG тАФ Agent Orientation

Read this file first when working on this codebase.

## What Is This Project?

StoryRPG is an interactive fiction app that blends visual-novel storytelling with tabletop RPG mechanics. Stories are generated by a multi-agent AI pipeline (text, images, audio) and played back as full-screen, image-backed reading experiences where the player makes choices that shape their character's identity. Think "choose your own adventure" with AI-generated prose, illustrations, and optional voice narration.

**Key design principle:** The player never sees stats, dice rolls, or numbers. All mechanics are expressed through narrative prose ("fiction-first").

## Architecture Overview

The system has three execution zones:

1. **Client app** (Expo) тАФ UI screens, story playback engine, Zustand stores. Runs on web, iOS, and Android.
2. **Proxy server** (Express on Node.js, port 3001) тАФ Proxies LLM/image/audio API calls (avoids CORS, keeps keys server-side), serves generated story files, manages generation job state.
3. **AI pipeline** (TypeScript, runs in worker process or in-browser) тАФ Multi-agent story generation using specialized agents, validators, and orchestration pipelines.

Data flow:
```
User triggers generation тЖТ Proxy тЖТ Worker тЖТ AI Pipeline (LLMs, image/audio APIs)
                                                тЖУ
Validated story JSON + images written to generated-stories/
                                                тЖУ
Client reads story files тЖТ Story Engine тЖТ Player experience
```

## Data Model Summary

The core hierarchy (defined in `src/types/index.ts`):

```
Story
 тФЬтФАтФА metadata, initialState, npcs[]
 тФФтФАтФА episodes[]
      тФЬтФАтФА scenes[]
      тФВ    тФЬтФАтФА beats[] (narrative units with text, images, audio)
      тФВ    тФФтФАтФА choices[] (with conditions, consequences, stat checks)
      тФФтФАтФА startingSceneId, unlockConditions, onComplete

PlayerState
 тФЬтФАтФА attributes, skills, relationships, flags, scores, tags
 тФЬтФАтФА inventory[], pendingConsequences[]
 тФФтФАтФА identityProfile, progress pointers
```

Choices can have: conditions (flag/stat gates), consequences (flag/stat changes), stat checks (fiction-first resolution), and branching targets.

### Story Circle Structure (load-bearing)

The pipeline honours a season-long Story Circle spine:

- `SourceMaterialAnalysis.anchors` тАФ `{ stakes, goal, incitingIncident, climax }`, inferred by `SourceMaterialAnalyzer` when not explicit in the source.
- `SourceMaterialAnalysis.storyCircle` тАФ `{ you, need, go, search, find, take, return, change }`.
- `SourceMaterialAnalysis.episodeBreakdown[i].storyCircleRole` тАФ which Story Circle beat(s) each episode carries.
- `SeasonPlan.anchors` + `SeasonPlan.storyCircle` carry these forward; `SeasonPlan.episodes[i].storyCircleRole` drives per-episode pressure, branch placement, cliffhanger handoff, and downstream agent prompts.
- `StoryCircleCoverageValidator` runs against `SeasonPlan` and blocks incomplete, non-contiguous, or out-of-order season spines.
- Downstream agents accept optional `seasonAnchors`, `seasonStoryCircle`, `episodeStoryCircleRole`, and `episodeCircle` inputs. The shared prompt helper `buildStructuralContextSection` in `src/ai-agents/prompts/storytellingPrinciples.ts` renders these into a consistent block for every agent that cares about narrative structure.

## Conventions and Patterns

- **Endpoints centralized** тАФ All URLs in `src/config/endpoints.ts`. Never hardcode URLs elsewhere.
- **Fiction-first** тАФ No visible stats/numbers in UI. All mechanics expressed narratively.
- **LLMs write, deterministic systems enforce** тАФ Deterministic code must never author or inject reader-facing prose. It enforces facts, criteria, and rules; when it detects a problem it gets the LLMs to rewrite (retry-with-feedback loops, final-contract repair handlers), never takes over the writing. Run-survival placeholders are permitted only when the exact string is registered in `src/ai-agents/constants/syntheticFallbackProse.ts` (choice outcome tiers: `choiceTextFallbacks.ts`) so `RouteContinuityValidator` / `OutcomeTextQualityValidator` block them at the final contract and route an LLM re-author. `src/ai-agents/validators/deterministicProseNeverShips.test.ts` enforces registry completeness for the known deterministic producers.
- **Minimal diffs** тАФ Keep code changes tightly scoped to the requested task. Do not reformat files, reorder imports, rename symbols, refactor unrelated code, or update generated artifacts unless required for the fix. Prefer the smallest clear patch that preserves existing style and behavior.
- **Node polyfills** тАФ `fs`/`path`/`process` are shimmed via babel and metro config so pipeline code can run in RN/web bundles.
- **Typecheck split** тАФ Shared app/test/contracts/worker configs plus target-specific `tsconfig.reader.json` and `tsconfig.generator.json`, all extending the base `tsconfig.json` as appropriate.
- **Tests** тАФ Co-located `*.test.ts` files, run with Vitest in Node env with RN stubs in `test/stubs/`.
- **Generated content** тАФ `generated-stories/` contains large JSON + image files and dominates repo size. Runtime artifacts (`.generation-jobs.json`, `pipeline-memories/`, `.ref-images/`) are transient.
- **EXPO_PUBLIC_** prefix тАФ Required for env vars that need to be accessible in the Expo client bundle. **Never use it for provider API keys or other secrets** тАФ Expo inlines `EXPO_PUBLIC_*` into the public reader bundle. Provider keys are server-side (proxy) only; `npm run verify:reader` scans the built bundle for leaked key values. (PostHog `phc_` publishable keys are the one client-safe exception.)
